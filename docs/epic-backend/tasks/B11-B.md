- **Estado:** 🟡 in-progress [owner: chat-2026-05-05, started: 00:00]
- **Por qué:** Slice vertical de **audit log e-2-e**: enforcement DB (trigger), writers (helper invocado desde Server Actions críticas) y reader (panel admin). Diseñar las tres juntas asegura que el contrato de cada entrada sea consistente entre quien escribe, quien protege la inmutabilidad, y quien lee. Separadas, una versión vieja del reader no entiende campos que un writer nuevo agrega, o el trigger bloquea un patrón que el writer asume permitido.
- **Entregable:**
  1. **Trigger append-only (DB):** migración con trigger `before update or delete on audit_log raise exception`. RLS ya restringe anon/authenticated, pero `service_role` lo saltea — el trigger lo blinda contra bugs propios.
  2. **Writer helper (TS):** `shared/domain/audit-log.ts` exporta `writeAuditEntry(actor, action, subject, payload)` invocado desde las Server Actions críticas (transiciones de orders en B9-B/B10-C, acciones admin en B11-A, suspend/reactivate en B11-C). Parte de la misma transacción que la acción.
  3. **Reader (panel):** `features/admin-audit-log/` consume `AuditLogRepository.find(filters)` con paginación keyset. Tabla readonly con filtros (actor, action, subject, rango de fechas).
- **Archivos:** `supabase/migrations/<ts>_audit_log_immutability.sql`, `shared/domain/audit-log.ts`, integración en Server Actions de B9-B/B10-C/B11-A/B11-C, `features/admin-audit-log/**`.
