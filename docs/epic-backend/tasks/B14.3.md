- **Notas:**
  - Convención adoptada: commits que tocan `supabase/migrations/` (o cambios SQL relevantes) usan `type=db`. release-please los rutea a la sección `Database` del CHANGELOG.
  - Scope expandido respecto al plan original: el entregable se hizo con `release-please-config.json` + `commitlint.config.cjs`. `config-conventional` tiene lista cerrada de types y bloqueaba el husky pre-commit con `db: ...` si solo tocábamos release-please. Se extendió `type-enum` para destrabarlo.
  - Verificación manual sugerida (post-merge a main): un commit `db: <msg>` aparece bajo `### Database` en el PR de release que abre release-please-action.
  - **Vitest preexistente rojo** (9 tests / 13 archivos) descubierto al correr la verificación combinada — no causado por este diff (config + docs only, vitest no carga estos archivos). Registrado en `docs/NEXT-TASK.md` como NT-30.
  - **Decisión versionado:** `db` aparece en `changelog-sections` sin override de `bump`. Por defecto release-please solo bumpea con `feat` (minor) y `fix`/`perf` (patch). Un commit `db: ...` aislado **no** dispara release; solo aparece en el CHANGELOG cuando ya hay un release abierto por otro tipo. Es el comportamiento deseado para migraciones en MVP — no queremos bumpear versión por un cambio de schema. Si en el futuro se quiere lo contrario, agregar `"bump": "patch"` a la entrada `db`.
