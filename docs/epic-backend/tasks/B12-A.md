# B12-A — Observability: structured logging con request IDs + slow query alerts

> **Fase:** B12 — Observability backend
> **Goal de la fase:** Instalar la visibilidad mínima para diagnosticar y alertar sobre problemas de performance / errores.
> **Acceptance criteria de la fase:** Panel admin permite leer top slow queries de `pg_stat_statements`; Sentry recibe queries >100ms mean y errores con request ID propagado; logs server-side son estructurados con request IDs.

> **Atajos:** [INDEX](../INDEX.md) · [convenciones](../convenciones.md) · [portabilidad](../portabilidad.md) · [decisiones](../decisiones.md)

- **Estado:** 🟡 in-progress [owner: chat-2026-04-28, started: 11:07]
- **Por qué:** Cierra la fase B12 con un slice operativo: structured logging (request IDs propagados) + alertas (Sentry recibe los slow queries detectados por el panel B12.1). Sin request IDs, las alertas son anónimas y dificultan la correlación con el log del request original — por eso conviene diseñar las dos cosas juntas.
- **Entregable:**
  1. **Structured logging:** middleware que inyecta `X-Request-Id` header; helper `logger.withRequestId(req)` integrado en Route Handlers de B7/B8/B12 y en Server Actions críticas.
  2. **Slow query alerts:** cron `/api/cron/check-slow-queries` cada 5 min que lee `pg_stat_statements`, compara con baseline, y manda breadcrumb/event a Sentry si un query cruzó el umbral (100ms mean o +50% vs semana pasada). El payload del evento incluye el request ID si está disponible.
  3. Schedule del cron incorporado a la migración de B7.1 vía patch.
- **Archivos:** `middleware.ts` (patch), `shared/services/logger.ts` (patch), `shared/utils/server-logger.ts` (patch si aplica), `app/api/cron/check-slow-queries/route.ts`, `supabase/migrations/<ts>_observability_alerts.sql` (patch a schedule_crons).
